程式架構
=============

簡介
------

FRC的程式主要有兩種架構，計時模式(Timed Based)跟指令模式(Command Based)，這兩個方式各有好壞，各自適合不同的情境

================== =========== =============
比較項目           Timed based Command Based
================== =========== =============
程式長度           短           長
專案複雜度         低           高
(大程式)除錯難易度 高           低
容易爆氣的程度       高          低
常見場景            早期測試   完整機器的扣
================== =========== =============

.. note::
    有很多強隊的code都會丟到OA(Open Alliance)上，不過6328他們的code的集成度高到抽象的地步了不建議看

Timed Based
------------

最基本也是最直覺的程式架構，就連Command Based也是從他延伸出來的，他把所有的東西都集成在一個檔案裡面，這樣的好處就是在小專案的時後(例如有人叫你在半小時後幫牠寫一個KOP的程式)可以快速的開發，壞處也是因為它全部的東西都集成在同一個檔案裡面，這會讓程式在肥起來之後的Debug難度會成幾何級數成長

**小結**

Timed Based的好處

- 開發快速
- 架構簡單
- 直覺明瞭

Timed Based的壞處

- 在大專案不容易Debug
- 隔天自己看不懂昨天的自己寫什麼鬼
- 別人要共編會比較難

Command Based
-------------

Command Based相比於Timed Based來說，他把一個程式拆成很多個檔案，並用一些概念去操作這些東西

I. 子系統
+++++++++

子系統(Subsystem)對應到機構的概念，一個子系統可以是Drivetrain, Intake, Shooter或者是Climber等等，就以實際的機器來說，一個subsystem可能會包含很多東西(例如8顆馬達4顆控制器跟一個陀螺儀)，也可能只會管一個馬達，透過把一台機器拆成很多個子系統有一些好處，例如

- 專案容易管理：今天如果有一個機構不會動了(例如intake)，那要去找問題解決會比用Timed Based容易很多，因為把每個子系統拆成一個檔案甚至是一個資料夾，所以只要去那個資料夾裡面看發生什麼事就好了不用順便做視力檢查
- 進度容易控制：分子系統的另外一個好處就是因為是獨立的檔案跟有的沒的，那有寫沒寫就看得出來，程式組有沒有在混都被看得一清二楚(咦？)
- 幹譙人比較容易：如果是用GitHub來協作的話，每一個code commit :del:`理論上` 都會經過PR才會進到主程式裡面，那誰寫的扣一清二楚。

II. 指令
+++++++++


指令(Command)是一個可以跑的東西 :del:`幹話` ，一個子系統可以透過指令來做到他要做的事情

III. 舉例
+++++++++

- Intake
    - 馬達: :code:`PivotMotor`, :code:`RollerMotor`
    - 原始架構
        - :code:`pivotIntake(Angle)` (搖得話可以用重複兩個值來達成)
        - :code:`intakeBalls(LinearVelocity)`
    - 實際上
        - :code:`intake(LinearVelocity)`
        - :code:`shake(Frequency)`

.. note::
    通常會需要高速旋轉的機構會因為轉動慣量的關係需要加底速，這個部分可以用獨立的private command去做就好了，算相對簡單的東西

- 360 turret
    - 馬達: :code:`FacingMotor`, :code:`HoodingMotor`, :code:`ShootingMotor`
    - 原始架構
        - :code:`turnTo(Rotation2d)`
        - :code:`pitchTo(Angle)`
        - :code:`shoot(LinearVelocity)`
    - 實際上
        - :code:`prepare()` 就讓他沒有輸入，資料要從api backend直接抓會比較可信任
        - :code:`shoot(LinearVelocity)`  這個會設default command，通常是設定球不會飛出去的速度

.. important::
    因為shooter這邊 :code:`prepare` 跟 :code:`shoot` 的指令會要同時跑，又一個子系統不會讓你一次跑兩個指令，所以這時候一個偷吃步就是會把其中一個的requirements拔掉，default command一定不能拔以外就看剩下幾個哪一個對安全性的影響比較小就拔哪個
